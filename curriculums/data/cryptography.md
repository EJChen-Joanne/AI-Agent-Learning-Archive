### 密碼學（Cryptography）

Web3 裡的「帳戶」「簽名」「地址」「所有權」都建立在密碼學之上。你不需要先成為密碼學研究者，但必須知道哪些動作是在證明身份，哪些動作是在授權資產，哪些東西一旦丟失就無法恢復。

#### 為什麼要學這個

很多 Web3 新手會把錢包理解成帳號系統，把簽名理解成登入確認，把地址理解成用戶名。這樣學到後面會很危險，因為鏈上系統沒有傳統互聯網那種中心化客服和密碼找回。

密碼學在這裡不是抽象理論，而是產品邊界：私鑰控制資產，簽名表達授權，雜湊固定數據，Merkle Tree 讓大量數據可以被高效驗證。

理解密碼學基礎，是為了知道什麼時候不能「點確認」。

這部分不要求你手推橢圓曲線公式。更現實的目標是：看到錢包彈窗、簽名請求、交易雜湊、Merkle proof 或合約驗證時，能判斷它們分別在證明什麼。

#### 第一性原理

鏈上身份不是由平台發給你的，而是由你能否證明自己控制某個私鑰決定的。
Web3 的帳戶系統把「誰有權限」從中心化數據庫轉成了可驗證的數學關係。好處是開放、可組合、無需平台許可；代價是用戶和應用都要承擔更明確的金鑰與簽名風險。

* 私鑰是控制權：丟失私鑰通常意味著失去控制權，洩漏私鑰意味著別人獲得控制權。
* 簽名是授權證據：簽名內容必須能被人讀懂，不能只展示一串難懂數據。
* 雜湊是承諾：雜湊不能還原原文，但可以用來驗證數據有沒有被改過。

#### 知識節點

**Hash**

Hash 函數會把輸入數據映射成一段固定長度輸出。理想情況下，哪怕輸入只改一個字符，輸出也會完全不同。鏈上常用雜湊來標識交易、區塊、數據承諾和合約字節碼。

雜湊不是加密。它通常不能被「解密回原文」，用途是驗證同一性和完整性。比如你拿到一份合約源碼，可以通過雜湊或編譯結果去檢查它是否和鏈上部署代碼一致。

常見用途包括：

* 交易雜湊：定位一筆交易。
* 區塊雜湊：定位一個區塊。
* 合約字節碼雜湊：檢查部署代碼是否一致。
* 數據承諾：先公布雜湊，之後再公開原文讓別人驗證沒有被改。

**Public Key**

Public Key 是和私鑰成對出現的公開資訊。別人可以用公鑰驗證某個簽名確實來自對應私鑰，但不能從公鑰反推出私鑰。

在以太坊語境裡，用戶常看到的是地址，而不是完整公鑰。地址可以理解成從公鑰進一步處理得到的短標識。地址可以公開，私鑰絕不能公開。

地址不是身份本身，只是一個可驗證控制關係的標識。某個地址「看起來像官方地址」沒有意義，真正重要的是它是否來自可信來源、是否和合約文檔、官網、區塊瀏覽器驗證資訊一致。

**Private Key**

Private Key 用來生成簽名，從而證明你控制某個帳戶。傳統互聯網裡密碼洩漏後可以找回或重置，但 Web3 裡私鑰一旦洩漏，攻擊者可以直接轉走資產或授權惡意合約。

所以錢包、助記詞、硬件錢包、多簽和帳戶抽象，本質上都在解決同一個問題：如何讓金鑰控制權更安全、更可恢復、更適合真實用戶。

私鑰管理的基本規則：

* 不截圖。
* 不上傳雲端。
* 不貼給網頁。
* 不發給 AI 工具或客服。
* 不放進代碼倉庫和 .env.example。
* 不用主錢包測試陌生 dApp。

**相關 topic**

```錢包（Wallet）```：繼續看私鑰如何被錢包管理，以及用戶如何發起簽名和交易。

**Signature**

Signature 由私鑰對消息生成。驗證者可以用公鑰或地址檢查簽名是否有效，從而確認這段授權確實來自對應帳戶。

簽名最容易出問題的地方是用戶看不懂自己簽了什麼。產品應該盡量展示可讀內容，例如簽名目的、域名、鏈 ID、合約地址、額度、過期時間和風險提示。對 Agent 來說，簽名更要有權限邊界，不能讓模型隨意觸發高風險授權。

**Merkle Tree**

Merkle Tree 會把很多數據逐層雜湊，最後得到一個 root。只要 root 被可信記錄下來，某個用戶就可以用 Merkle proof 證明「我的數據在這批數據裡」，而不用下載全部數據。

它常見於空投名單、輕客戶端、狀態證明和可驗證數據結構。理解 Merkle Tree，有助於理解為什麼區塊鏈可以把大量狀態組織成可驗證結構。

#### 在 AI x Web3 中的位置

AI Agent 如果要解釋交易、判斷授權、生成鏈上操作或幫助用戶管理權限，就必須理解簽名和金鑰的邊界。模型可以解釋「這筆授權看起來在批准某個 token 額度」，但不能替用戶保管私鑰，也不能在用戶不理解內容時推動簽名。

更進一步，AI 輸出本身也可能需要被簽名、雜湊或記錄，作為後續審計的證據：某個 Agent 在什麼輸入下給出了什麼建議，用戶是否確認，執行結果是否和建議一致。

#### 最小實踐

做一個簽名觀察練習：

* 創建一個測試錢包，不放真實資產。
* 在測試環境裡發起一次普通轉帳或消息簽名。
* 在區塊瀏覽器或錢包界面裡觀察地址、交易雜湊、簽名提示和鏈 ID。
* 寫下哪些資訊是可以公開的，哪些絕不能公開。
* 比較「簽名消息」和「發送交易」的區別。

不要用主錢包做練習，不要把助記詞或私鑰貼到任何網頁、聊天工具或代碼倉庫。
