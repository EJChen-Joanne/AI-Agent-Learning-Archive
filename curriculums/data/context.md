### 上下文（Context）

Context 是模型這一次能看到、能使用、能被影響的信息空間。真正難的不是把更多內容塞進去，而是把系統規則、用戶目標、歷史狀態、工具結果和外部文檔分清楚。

#### 為什麼要學這個

LLM 沒有自動連接你的資料庫、文件系統、API、用戶歷史和項目文檔。它只會基於當前上下文生成輸出。如果上下文錯了、缺了、過期了，模型再強也會給出不可靠答案。

很多 AI 應用的問題表面看是「模型不夠聰明」，實際是 context 設計混亂：把不可信網頁當系統指令，把舊文檔當最新規則，把用戶願望當事實，把上一次任務的狀態洩漏到下一次任務。

在任何帶工具調用的場景裡，這都會直接影響安全。模型如果看不到真實工具結果，就只能猜；Agent 如果不知道當前狀態、權限和預算，就可能生成越界操作。

#### 第一性原理

模型只能基於它看見的上下文行動；系統必須決定什麼能進上下文、帶著什麼身份進去、過期後怎麼退出。
Context 不是簡單的長文本拼接，而是一套信息治理問題。你要為每類信息標注來源、時效、權限和可信度。否則模型會把「用戶說的」「網頁寫的」「鏈上查到的」「系統規定的」混在同一層處理。

可信來源要顯式標注：系統狀態、用戶輸入、檢索文檔、工具結果要分區放置。
上下文要可刷新：狀態、配置、權限、價格、版本和任務進度不能長期緩存後繼續使用。
記憶要可撤銷：用戶偏好和歷史任務可以提升體驗，但不能變成隱藏權限或永久身份假設。

#### 知識節點

**Context Window**

```Context Window``` 是模型一次請求能處理的最大上下文範圍。窗口越大，能放進來的資料越多，但不代表模型會完美使用每個細節。

長上下文常見的問題是「看見了但沒抓住重點」。如果你把合約源碼、審計報告、治理討論和交易日誌全部塞進去，模型可能被無關內容分散注意力，也可能引用過期段落。

在實際產品裡，context window 應該配合檢索、摘要和結構化數據使用。關鍵狀態直接給，長文檔按需取，低可信內容隔離標注。

**Context Engineering**

*```Context Engineering``` 是設計上下文進入模型的方式*。它包括選擇數據源、排序、裁剪、標注來源、隔離不可信內容、決定哪些信息必須每次重新查詢。

一個穩定的工具型 Agent 上下文，通常不只是「用戶問題 + 一段 JSON」。它還應該包括：

* 當前任務狀態
* 工具返回結果
* 相關日誌或證據
* 可信數據來源
* 外部檢查結果
* 用戶原始意圖
* 系統禁止事項和輸出 schema

Context Engineering 的目標不是塞滿窗口，而是讓模型在正確的信息層級裡工作。

**相關Topic**

```Chain-aware Context```：繼續看鏈上狀態如何進入 Agent 上下文。
```RAG```：當上下文來自大量文檔時，需要檢索和引用機制。

```Memory```

*```Memory``` 是跨請求保留的信息*，例如用戶偏好、歷史任務、常用錢包、項目配置、上次分析結果。

Memory 可以讓 Agent 更順手，但也容易帶來隱性風險。比如系統記住「用戶常用 0xabc 地址」，下次就默認它是收款地址；或者記住「用戶風險偏好較高」，就放鬆了高風險交易確認。

Memory 不能替代實時授權。用戶過去允許某個動作，不代表現在仍然允許。所有涉及身份、權限、資產或外部副作用的記憶，都必須重新綁定當前會話和當前授權。

```Knowledge Base```

```Knowledge Base``` 是系統可檢索的外部知識庫，比如產品文檔、SDK 文檔、代碼說明、論壇討論、FAQ、內部 runbook。

它適合解決模型知識過期的問題，但不會自動保證正確。知識庫需要維護來源、更新時間、版本、適用網路和廢棄狀態。否則 RAG 只會把舊錯誤更穩定地送進模型。

對 builders 來說，知識庫至少要記錄：

* 文檔來源和 URL
* 最後更新時間
* 適用協議版本
* 適用版本或運行環境
* 是否來自官方或第三方
* 是否需要人工複核

#### 在 AI x Web3 中的位置

Context 是連接模型和現實系統的入口。AI 負責推理，Web3 負責狀態和結算；context 決定模型看到的是用戶幻想、過期文檔，還是可驗證的鏈上事實。

一個靠譜的 Agent 上下文通常會分層：

* 指令層：系統規則、工具使用規則、禁止事項。
* 任務層：用戶目標、本次會話參數。
* 事實層：鏈上狀態、工具結果、simulation。
* 知識層：文檔、標準、論壇、歷史案例。
* 記憶層：用戶偏好和項目配置。

層級越清楚，後續越容易做權限控制、prompt injection 防護和審計。

#### 最小實踐

為「錢包授權檢查 Agent」設計一份 context spec。

選擇一個場景：用戶問「這個 dApp 要我 approve，可以簽嗎？」你需要列出模型回答前必須拿到的上下文：

* chain id 和當前區塊
* token 合約、spender 地址、approve 數量
* 用戶當前 allowance 和餘額
* spender 是否在可信列表
* simulation 或靜態檢查結果
* dApp 頁面提供的說明，標記為不可信外部內容
* 用戶本次意圖

再寫清楚哪些字段必須實時查詢，哪些可以來自緩存，哪些不能被模型當成事實。
