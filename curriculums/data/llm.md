### 大語言模型（LLM）

LLM 不是一個會聊天的黑盒，而是把大量文本、代碼和多模態信號壓進參數裡的概率模型。先理解它如何處理 token、上下文、模式和不確定性，後面才知道什麼時候該信它、什麼時候必須驗證它。

#### 為什麼要學這個

很多 AI 應用一開始都會說"我們用大模型"。這句話還不夠。你需要繼續追問：模型負責理解用戶意圖，還是生成內容？它只是在回答問題，還是能調用工具？它的錯誤會停留在文本裡，還是會進入真實工作流？

LLM 是今天大多數 AI 應用的能力底座。它能總結文檔、寫代碼、生成計劃、抽取結構化信息，也能把自然語言轉成更適合程序處理的格式。但它本身不是數據庫，不會自動知道外部世界的最新狀態，也不能替系統承擔事實校驗。

學 LLM 的目標不是背模型參數，而是建立一個判斷：模型輸出是候選結果，不是事實本身；模型能力是推理入口，不是最終驗證。

#### 第一性原理

LLM 生成的是概率上合理的輸出，而不是天然可信的事實。
這決定了 LLM 在任何嚴肅系統裡的位置：它可以幫你理解、歸納、生成和規劃，但不能單獨承擔事實來源、權限判斷和最終執行。越靠近真實動作，越需要外部數據、確定性規則和人工或系統校驗。

把模型當推理層，不當真相源：關鍵事實必須來自數據庫、API、日誌、文檔源或其他可信系統。
把輸出變成可檢查對象：摘要、分類、計劃和操作建議都應該盡量落成 schema、參數、引用或日誌，而不是只停留在自然語言。
把不確定性前移暴露：模型不知道、資料過期、上下文不足時，要讓系統顯式降級，而不是編一個順滑答案。

#### 知識節點

**Token**

```Token``` 是模型處理文本的「基本單位」。它不一定等於一個漢字、一個英文單詞或一個符號，而是 **tokenizer 切分後的segment**。

Token 直接影響三件事：上下文能放多少、調用成本是多少、模型能不能完整看見關鍵信息。長文檔、代碼、JSON、日誌和多輪對話都很容易把上下文塞滿。你需要決定哪些信息原樣放入，哪些先壓縮，哪些交給檢索系統按需取回。

不要把"頁面很短"誤認為"token 很少"。代碼、JSON、長標識符、表格和混合語言文本經常比普通段落更吃token。

**Embedding**

*```Embedding``` 是把文本、代碼或其他對象映射成向量*，用來衡量"語義上是否接近"。它常用於*搜索、聚類、推薦、異常檢測和RAG*。

Embedding 適合幫你從文檔、知識庫、代碼倉庫、討論紀錄和產品日誌中找相關材料。但它不適合單獨判斷"這個結論是否正確"。向量相似度只能說明內容接近，不能替代來源校驗、規則檢查和人工判斷。

**相關Topic**

```RAG```：繼續看 embedding 如何進入檢索增強生成系統。
OpenAI Embeddings Guide：理解 embedding 的用途、向量距離和 API 形態。

```Transformer```

Transformer 是現代 LLM 的核心架構之一。它的關鍵能力來自 *attention*：模型可以在生成時關注輸入中的不同位置，學習詞、代碼、事實和上下文之間的關係。

你不需要一開始就推公式，但要理解一個工程事實：Transformer 擅長在上下文裡找模式，不等於它擁有穩定數據庫。它能把文檔、代碼和用戶目標組合起來生成解釋，也可能因為上下文缺失或相似模式誤導而給出錯誤歸納。

Transformer 給了模型強大的模式組合能力，但沒有給它事實最終裁決權。

```Hallucination```

Hallucination 指模型生成了看起來合理、但並不真實或無法驗證的內容。它可能編造API、錯誤解釋代碼、引用不存在的資料，或者把舊版本文檔當成當前狀態。

在普通問答裡，幻覺可能只是答案質量問題。在任何帶執行能力的系統裡，幻覺都會變成流程風險：錯誤參數、錯誤權限解釋、錯誤操作建議，都可能進入後續自動化鏈路。

處理幻覺不要只靠"寫更好的 prompt"。更可靠的方式是*把模型輸出接到外部校驗*：來源引用、schema 校驗、規則檢查、人工確認和審計日誌。

```Evaluation```：用測試集和回歸評估持續捕捉模型錯誤。

```AI Security```：看模型錯誤和攻擊輸入進入工具執行後的風險。

```Multimodal```

Multimodal 模型可以處理文本、圖片、音頻、視頻或屏幕截圖。對 builders 來說，它的價值不是"更炫"，而是讓模型讀懂更多真實工作界面：圖表、控制台、設計稿、應用頁面、錯誤截圖和確認彈窗。

但多模態輸入同樣需要邊界。截圖裡的文字可能被遮擋，圖表可能缺少坐標，頁面可能被偽造。模型可以輔助識別和解釋，但關鍵判斷仍要回到結構化數據和可信來源。

#### 在 AI x Web3 中的位置

LLM 位在 AI x Web3 系統的理解和生成層。它負責把用戶目標轉成可討論的計劃，把複雜鏈上數據解釋成人能讀的語言，把文檔和代碼串成可執行思路。

真正的產品通常還需要這些層配合：

* 數據層：RPC、索引器、預言機、向量庫、項目文檔。
* 編排層：Prompt、Context、RAG、Agent workflow。
* 執行層：工具調用、錢包、Smart Account、合約交互。
* 安全層：Guard、simulation、權限策略、人工確認、日誌。
* LLM 越靠近執行層，系統越要把它的自然語言輸出變成可驗證對象。

#### 最小實踐

做一個"交易解釋器"的最小版本。

輸入一筆交易哈希，讓系統讀取交易詳情、事件日誌和相關合約 ABI，然後讓 LLM 生成一段解釋。要求輸出包含：

用戶發起了什麼動作、涉及哪些資產和地址、哪些信息來自鏈上數據，哪些是模型推斷、模型不確定的地方、如果要簽類似交易，用戶應該檢查什麼、練習重點不是讓解釋很漂亮，而是把 模型生成、鏈上事實、來源邊界、不確定性 分開。
