### 推理服務（Inference）

訓練決定模型學到了什麼，推理決定模型在真實產品裡如何響應用戶。對 builder 來說，Inference 不是一個雲服務按鈕，而是延遲、成本、上下文、穩定性和部署邊界的綜合選擇。

#### 為什麼要學這個

很多 AI 應用的第一個 demo 都是直接調用模型 API。但產品一旦進入真實用戶、真實流量和真實任務，問題會立刻變具體：響應太慢怎麼辦，成本怎麼控制，模型換了會不會破壞輸出，隱私數據能不能送到第三方 API，本地模型夠不夠用，服務崩了怎麼降級。

Inference 這一層決定了模型能力如何被產品消費。它連接上游的模型、prompt、RAG、Agent，也連接下游的接口、隊列、緩存、監控和用戶體驗。

模型能力只有被穩定、可控、可觀測地調用，才算進入了產品系統。

#### 第一性原理

推理服務的核心不是「跑出答案」，而是在約束條件下交付可用答案。
同一個模型，在 API、GPU 服務、本地量化模型和邊緣設備上的表現會完全不同。你選擇的不是單一模型，而是一套延遲、成本、質量、隱私和運維複雜度之間的平衡。

質量有代價：更強模型通常意味著更高成本、更長延遲或更複雜部署。
部署改變邊界：API model 讓你少管基礎設施，本地模型讓你多拿控制權。
服務要可替換：把模型調用封裝清楚，才有機會做 fallback、灰度和評估。

#### 知識節點

**API Model**

難度：初級。先理解託管模型 API 的調用方式、參數、限制和成本結構。

API Model 指通過雲端服務調用模型，例如把輸入發送給模型提供方，再接收文本、結構化輸出、工具調用結果或多模態結果。它的優勢是上手快、模型更新快、基礎設施負擔低。

但 API 不等於「沒有工程問題」。你仍然要處理速率限制、超時、重試、日誌脫敏、帳單控制、版本變更和輸出回歸。尤其是 Agent 場景，一次用戶請求可能觸發多輪模型調用，成本和延遲會被放大。

**相關 topic**

```提示詞（Prompt）```：API 調用的輸入質量會直接影響輸出穩定性。
```評估（Evaluation）```：模型或參數變更前，需要用評估集檢查輸出是否退化。

**Local Model**

難度：中級。你需要理解本地運行模型為什麼更可控，也為什麼更吃硬件、部署和調參能力。

Local Model 是在自己的設備、服務器或私有環境中運行模型。它適合隱私要求高、成本敏感、需要離線運行或需要深度定制的場景。

本地模型的限制也很直接：模型權重、顯存、上下文窗口、並發能力、量化方式、推理框架都會影響效果。對很多團隊來說，本地模型不是替代所有 API，而是承擔特定任務：分類、抽取、代碼補全、輕量 Agent、隱私數據處理或 fallback。

**相關 topic**

```微調（Fine-tuning）```：本地或私有模型常常和微調、LoRA、領域數據一起出現。

```Quantization```

難度：中級。量化要在模型體積、推理速度和輸出質量之間做取捨，不能只看「能不能跑起來」。

Quantization 是把模型權重或計算精度降低，例如從 FP16 降到 INT8、INT4，用更少顯存和計算資源運行模型。它讓個人電腦、小 GPU 或邊緣設備運行大模型成為可能。

問題在於，量化可能降低輸出質量，尤其是長推理、代碼生成、多語言、數學和工具調用場景。判斷量化模型是否可用，不應該只靠主觀聊天體驗，而要用你自己的任務樣本測試。

```Serving```

難度：高級。Serving 關注的是把模型變成穩定服務：並發、隊列、流式輸出、緩存、監控、失敗處理和版本管理。

Serving 是把模型部署成可被應用調用的服務。它通常涉及模型加載、請求隊列、批處理、GPU 利用率、token streaming、日誌、指標、健康檢查和擴縮容。

一個成熟的推理服務至少要回答：

* 請求失敗時怎麼重試或降級？
* 模型版本怎麼灰度？
* 輸入輸出日誌如何脫敏？
* 長請求是否要進入隊列？
* 成本、延遲和錯誤率如何監控？

Serving 做不好，模型越強，線上問題越難排查。

#### 在 AI x Web3 中的位置

AI x Web3 項目經常需要把鏈上數據、用戶意圖、合約接口和風險提示送進模型。Inference 這一層會直接影響用戶能否安全地等待結果、是否願意支付成本、以及 Agent 是否能在合理時間內完成鏈上前置判斷。

如果模型要參與交易解釋、合約分析、策略建議或 Agent 執行，推理服務必須留下可審計記錄：用了哪個模型、輸入來自哪裡、輸出是什麼、是否觸發工具、失敗時如何處理。鏈上動作不可逆，推理層就不能像普通聊天一樣隨意。

#### 最小實踐

做一個最小推理對比：

* 選擇同一個任務，例如「總結一筆交易的風險」或「從合約 ABI 中提取可調用方法」。
* 用一個託管 API 模型跑一次。
* 用一個本地模型或較小模型跑一次。
* 記錄延遲、成本、輸出質量、隱私邊界和失敗情況。
* 寫出你會在產品中選擇哪一種，以及 fallback 怎麼設計。

重點不是證明哪個模型最好，而是學會把模型選擇放進真實約束裡。
